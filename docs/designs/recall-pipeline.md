# Design: Recall Pipeline Production Parity

Generated by /plan-ceo-review on 2026-03-19
Mode: SELECTIVE EXPANSION | Branch: main

---

## Problem

The eval harness (`evals/recall_eval.py`) implements a full recall pipeline
(HyDE + iterative retrieval + LLM reranker) that achieves strong recall on
the 124-case hard benchmark. The production code (`core/memory.py:make_recall`)
uses raw vector KNN only. The two paths have completely diverged.

Additionally, several supporting signals are broken:
- `access_count` is never updated → `_access_utility` is always neutral (0.5)
- `expansion_state` column exists in DB but no code uses it
- `MemoryQuery.kinds/labels` filters are silently dropped

---

## Approach: Clean port (Approach B)

Introduce proper Protocol ports for the new pipeline stages.
Port the eval wrappers into `core/memory.py` as proper `make_*` factories.
Store expansion embeddings in sqlite-vec for production use.

### New ports (ports.py)

```python
class RerankFn(Protocol):
    async def __call__(self, query: str, nodes: list[MemoryNode]) -> list[str]: ...

class RewriteQueryFn(Protocol):
    async def __call__(self, query: str) -> str: ...

class UpdateNodeFn(Protocol):
    async def __call__(self, ids: tuple[str, ...]) -> Result[None, NodeWriteError]: ...
```

`HydeFn` reuses `ExpandContextFn` — identical signature `(str) -> list[str]`.

### Pipeline stages (core/memory.py)

```
make_recall(search, embed, hydrate, update_access)
  └─ base recall + access_count update

make_hyde_recall(search, embed, doc_embed, hydrate, hyde_fn, update_access)
  └─ wraps make_recall with HyDE score blending

make_iterative_recall(base_recall, embed, search, hydrate)
  └─ wraps base with round-2 multi-hop search

make_reranked_recall(base_recall, rerank_fn, hydrate, decayed_ids)
  └─ wraps with LLM reranking over wider candidate pool

make_rewritten_recall(base_recall, rewrite_fn)
  └─ rewrites query to surface implicit knowledge before all stages
```

### Blend weights

```python
_W_HYDE = 0.15               # final = base*(1-W_HYDE) + hyde*W_HYDE
_RERANK_FETCH_FACTOR = 5     # fetch top_k*5 before reranking
_HYDE_TIMEOUT   = 5.0        # seconds, falls back to {} on timeout
_RERANK_TIMEOUT = 30.0       # seconds, falls back to vector order
_REWRITE_TIMEOUT = 10.0      # seconds, falls back to original query
```

**Removed:** `_W_EXP / _W_DOC` blend — reverted to MAX scoring per node_id.
Expansion (query-side) and primary (doc-side) embeddings coexist in the same
`vec_index`; `_dedup_scores` takes the MAX cosine per node. HyDE uses
`primary_search` (is_primary=1 only) to avoid cross-modal contamination.

### Full pipeline data flow

```
QUERY TEXT
    │
    ├─REWRITE (optional, timeout=10s):
    │   rewrite_fn(query) → rewritten_query
    │   fallback: original query on timeout
    │
    ├─[embed(rewritten_query)]─────────────→ q_emb
    ├─[search(q_emb, top_k×5)]────────────→ base_hits
    │
    ├─HyDE (parallel, timeout=5s):
    │   hyde_fn(query) → snippets
    │   asyncio.gather([doc_embed(s) for s in snippets]) → embs
    │   asyncio.gather([primary_search(e, top_k×5) for e in embs]) → hyde_hits
    │   → hyde_scores: dict[id, float]  (fallback: {} on timeout)
    │
    ├─BLEND: blended = base*(1-W_HYDE) + hyde*W_HYDE
    │
    ├─ITERATIVE (round 2):
    │   ctx = top3_content joined
    │   embed(query + ctx) → r2_emb → r2_hits
    │   merged = max(blended, r2) per node
    │
    ├─RERANK (timeout=30s):
    │   candidates = top_k×5 merged, decayed_ids stripped
    │   rerank_fn(query, candidates) → ranked_ids
    │   fallback on timeout: merged order
    │
    ├─[hydrate(top_ids)]
    ├─[update_access(top_ids)] — best-effort, log on fail
    └─→ RecallResult(nodes, scores)
```

### Expansion indexing

New CLI command `openhas expand`:
- Queries `nodes WHERE expansion_state='pending'`
- For each pending node: `expand_fn(content)` → queries, `query_embed(q)` → embeddings
- Stores embeddings in `vec_index` + `vec_meta` (multiple rows per node_id)
- Sets `expansion_state='expanded'` (or `'failed'` on error)
- `--concurrency N` (default 4): semaphore-limited parallel LLM calls

### Eval alignment

`evals/recall_eval.py` rewritten to use production `sqlite_vec_store` factories.
In-memory numpy index replaced with the same sqlite-vec path production uses.
Eval numbers now reflect actual production behavior.

## Error handling

Every LLM-backed stage has:
1. `asyncio.wait_for(timeout)` — hard timeout
2. `except Exception` with `logger.warning(...)` — log, don't crash
3. Fallback return value — base scores or original order

No LLM failure can propagate as a RecallError. Users always get results.
